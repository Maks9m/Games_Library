# Games_Library
4th semester .NET class labs2-6 project

## Ідея проєкту: "Бекенд-ядро ігрової платформи (Game Launcher Core)"

Проєкт імітуватиме роботу сервісу, який керує бібліотекою ігор, профілями гравців, їхніми досягненнями та налаштуваннями застосунку.

Ось як елегантно вписати всі лабораторні роботи в цю ідею:

1. Лабораторна №2: LINQ to XML

У методичці Варіант 20 ідеально підходить під нашу концепцію — це "гейм-платформа".

Структура даних: Тобі потрібно буде реалізувати сутності, де гравці грають в ігри , ігри мають розробників , розробники створюють кілька ігор , а досягнення відкриваються гравцями.
+3

Введення та збереження: Зробиш консольне меню для додавання нових ігор чи гравців, які будуть зберігатися у файл за допомогою XmlWriter.

Десеріалізація: Завантаження цієї бібліотеки при старті програми буде відбуватися через XmlDocument або XmlSerializer.

Запити (LINQ to XML): Реалізуєш 6 унікальних запитів. Обов'язково включи два запити з варіанту:

Пошук гравців, які відкрили найбільше досягнень, із середньою кількістю досягнень на гру.

Визначення розробників із найбільшою кількістю ігор з високим рейтингом.

2. Лабораторна №3: Робота з JSON

Якщо XML ми використовуємо як імпровізовану базу даних ігор та користувачів, то JSON ідеально підійде для файлу конфігурації лаунчера (наприклад, settings.json).

Що зберігати: Шляхи до папок завантаження, ліміти швидкості скачування, налаштування мови інтерфейсу, теми оформлення (світла/темна).

Завдання: Реалізувати клас AppConfig, який при старті зчитує settings.json, а при зміні налаштувань у консолі — серіалізує оновлені дані назад у файл.

3. Лабораторні №4-6: Патерни проєктування (Design Patterns)

Тут ми впровадимо 3 класичні патерни, які органічно впишуться в архітектуру:

* Singleton (Твірний патерн):

* Де використати: Для класу ConfigurationManager, який відповідає за роботу з JSON-налаштуваннями (ЛР №3).

* Чому: Лаунчер повинен мати лише один екземпляр налаштувань, доступний з будь-якого місця програми, щоб уникнути конфліктів при читанні/запису файлу settings.json.

Strategy (Поведінковий патерн):

* Де використати: Для сортування та фільтрації ігрової бібліотеки (обробка даних з ЛР №2).

* Чому: Гравці можуть хотіти сортувати ігри за різними критеріями (за рейтингом, за алфавітом, за датою виходу). Замість нагромадження if/else, ти створюєш інтерфейс ISortStrategy і класи SortByRating, SortByTitle, які викликатимуться через LINQ-запити.

Facade (Структурний патерн):

* Де використати: Для створення головного класу LauncherCore.

* Чому: В тебе є підсистема XML (база), підсистема JSON (налаштування) і логіка підрахунку статистики. Facade приховає цю складність і надасть консольному UI прості методи на кшталт launcher.Start(), launcher.GetTopPlayers(), launcher.ChangeSetting().
